# 记忆分类决策流程

## 核心原则

每条待记录信息必须经过分类判断，确定存储位置：
- **长期记忆**（`.codebuddy/MEMORY.md`）：跨会话持久有效的知识，下次会话仍需使用
- **近期记忆**（`.codebuddy/memory/YYYY-MM-DD.md`）：当前工作阶段的过程性记录，主要服务于短期连续性

## 分类决策矩阵【必须】

### 必须归入长期记忆（MEMORY.md）的信息类型

| 类别 | 具体内容 | 示例 |
|------|---------|------|
| **用户偏好** | 编码风格、命名习惯、技术选型偏好、工作流偏好 | "用户偏好组合而非继承"、"日志用 SLF4J 不用 Log4j" |
| **项目约定** | 分支策略、提交规范、代码审查要求、部署流程 | "所有改造点必须支持一键回退"、"开关命名格式 open_xxx" |
| **核心设计决策** | 已确认的架构选择、方案选型及理由 | "库存排序：新建独立服务类，不修改原类" |
| **业务规则** | 领域逻辑、业务约束、计算公式 | "增量/存量判断粒度：型号级别" |
| **架构认知** | 系统架构层次、模块职责、调用关系 | "排序在策略层，对冲在对冲层，两者分离" |
| **项目元信息** | 项目名称、分支用途、技术栈、团队角色 | "当前分支：优化库存排序算法" |
| **跨会话约束** | 适用于所有后续会话的硬性规则 | "老代码完全不动，新逻辑全部新建" |
| **否决记录** | 被明确否决的方案及原因（防止重蹈覆辙） | "❌ 方案B被否决：复用字段会导致语义不匹配" |

### 必须归入近期记忆（日志）的信息类型

| 类别 | 具体内容 | 示例 |
|------|---------|------|
| **当前会话进度** | 正在进行的任务状态、完成百分比 | "子任务1：10个疑问已确认 ✅" |
| **过程性讨论** | 分析推理过程、中间方案比较、排除逻辑 | "分析了3种方案，方案A适合因为..." |
| **代码改动清单** | 本次会话修改的文件列表和改动点 | "修改了 PnMatchStrategy.java 第59-62行" |
| **临时确认** | 会话中的中间确认，尚未形成最终决策 | "先按这个方向试试" |
| **调试记录** | Bug排查过程、错误原因、修复步骤 | "NPE 根因：stockList 未做空值检查" |
| **待办事项** | 短期任务列表、下一步计划 | "⏳ 下一步：开始编码改造点①" |
| **实施细节** | 具体的实现代码、配置变更、伪代码 | "改造伪代码：if (isOpen()) { ... }" |
| **工具/环境记录** | 临时的环境问题、工具使用记录 | "Maven 编译需要 -DskipTests" |

## 分类判断决策树【必须】

```
信息产生
    │
    ├─ 问自己："下一次全新会话启动时，这条信息是否仍然必要？"
    │
    ├─ 是（下次会话仍需要）
    │   │
    │   ├─ 信息是否会随时间变化？
    │   │   ├─ 不变（约定/规则/偏好） → 【长期记忆】原地更新
    │   │   └─ 会变（决策可能迭代） → 【长期记忆】原地更新 + 注明日期
    │   │
    │   └─ 是否已有相同主题的长期记忆？
    │       ├─ 有 → 更新已有条目（replace_in_file），不新增重复条目
    │       └─ 无 → 新增条目到对应分类区块
    │
    └─ 否（仅当前阶段有用）
        │
        └─ 追加到当日日志 → 【近期记忆】
```

## 双写规则：需要同时记录两处的场景

某些信息需要**同时**写入两处：

| 场景 | 近期记忆写什么 | 长期记忆写什么 |
|------|--------------|--------------|
| 用户确认核心方案 | 完整讨论过程 + 确认细节 | 方案结论 + 选择理由（精简版） |
| 发现关键架构认知 | 分析过程 + 代码引用 | 架构认知结论（去掉推理过程） |
| 用户否决方案 | 否决讨论完整记录 | ❌ 被否决方案 + 原因（一行即可） |
| 确认项目约定 | 讨论上下文 | 约定本身（规则化表述） |

**双写判断标准**：信息同时满足"当前工作需要详细过程"和"未来会话需要核心结论"。

## MEMORY.md 结构规范【必须 - 精简优先】

长期记忆按分区组织，每条目 ≤ 100 字，总条目 ≤ 30 条：

```markdown
# 项目长期记忆

## 项目概述
项目名/技术栈/分支用途，一到两句话。

## 用户偏好与约定
每条一行，规则化表述。

## 🔴 [最高优先级约束] (YYYY-MM-DD)
约束内容，一句话。

## [模块] - [决策] (YYYY-MM-DD)
核心结论，一到两句话。（禁止 ### 子标题展开）

## [模块] - ❌ [否决方案] (YYYY-MM-DD)
否决结论 + 原因，一句话。
```

**格式要求**：
- 每个 `##` 条目为独立单元，内容紧跟其后，不使用 `###` 子标题
- 已有同主题 → 原地更新，不新增重复条目
- 超限时使用 `/memory:tidy --compact` 整理

## 日志晋升机制

### 晋升触发条件

日志中的信息在以下条件下应**晋升**到长期记忆：

1. **同一决策被连续2天引用**：说明信息跨会话持久有效
2. **用户最终确认**：从"临时确认"升级为"最终决策"
3. **会话结束前整理**：主动扫描当日日志，提取符合长期记忆标准的信息

### 晋升操作

1. 从日志提取核心结论（去掉过程、推理、中间状态）
2. 精简为规则化表述（不超过3行）
3. 更新到 MEMORY.md 对应分区
4. 日志中原始记录**保留不动**（日志是完整历史，不删减）

### 日志清理

- 超过 30 天的日志文件可删除
- 删除前确保所有有价值信息已晋升到 MEMORY.md
