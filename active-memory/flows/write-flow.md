# 记忆写入流程

## 前置步骤：信息分类【必须】

**每次写入前必须先完成分类**（详见 `classify-flow.md`）：

核心判断："下一次全新会话启动时，这条信息是否仍然必要？"
- **是** → 走"长期记忆写入流程"（可能需要同时写日志，见双写规则）
- **否** → 走"近期日志追加流程"

### 需要双写的场景

| 场景 | 近期日志写什么 | 长期记忆写什么 |
|------|--------------|--------------|
| 用户确认核心方案 | 完整讨论过程 + 确认细节 | 方案结论 + 选择理由（精简版） |
| 发现关键架构认知 | 分析过程 + 代码引用 | 架构认知结论（去掉推理过程） |
| 用户否决方案 | 否决讨论完整记录 | ❌ 被否决方案 + 原因（一行即可） |
| 确认项目约定 | 讨论上下文 | 约定本身（规则化表述） |

## 近期日志追加流程【必须】

1. **确认分类结果**：信息属于近期记忆（或双写场景的近期部分）
2. **读取当日日志**：`read_file` 读取 `.codebuddy/memory/YYYY-MM-DD.md`（不存在则先创建目录和文件）
3. **追加新内容**：`replace_in_file` 追加新记录到文件末尾（不修改已有内容）
4. **继续任务**：记录过程不打断当前任务流程，不等用户确认

## 长期记忆写入流程【必须】

### 立即写入（信息确认时）

触发条件：用户确认方案/偏好/约定、否决方案、首次识别项目元信息

1. **读取 MEMORY.md**：`read_file` 读取 `.codebuddy/MEMORY.md`
2. **定位分区**：根据信息类别找到对应分区（项目概述/用户偏好/设计决策等）
3. **原地更新**：`replace_in_file` 更新对应分区（注明日期）
   - 已有同主题条目 → 更新已有条目，不新增重复
   - 无同主题条目 → 在对应分区末尾新增
4. **如需双写** → 同时执行"近期日志追加流程"

### 延迟写入（会话结束前）

触发条件：架构认知、未即时归档的设计决策

1. **扫描当日日志**：识别符合长期记忆标准的信息
2. **提取核心结论**：去掉过程、推理、中间状态，精简为规则化表述
3. **执行"立即写入"流程**

## 日志晋升机制

日志中的信息在以下条件下应晋升到长期记忆：
1. 同一决策被连续2天引用
2. 从"临时确认"升级为"最终决策"
3. 会话结束前整理时发现符合长期记忆标准

### 清理旧日志（可选）

删除 30 天前的日志文件，删除前确保有价值信息已晋升到 MEMORY.md。

## 记录格式【必须 - 精简优先】

### 核心原则

- **一条记录 ≤ 3 行**：主题行 + 1~2 行内容，禁止展开子标题
- **结论优先**：只记结论，不记过程/推理/讨论
- **状态标记必带**：✅（确认）❌（否决）⏳（待办）

### 当日日志格式 (YYYY-MM-DD.md)

```markdown
## [主题] - [简短描述]
- ✅ 结论/确认内容（一句话）
- ❌ 否决内容及原因（一句话）
- ⏳ 待办事项（一句话）
```

**示例**（正确 ✅）:
```markdown
## Redis 缓存方案 - 已确认
- ✅ Redis 7.0 做缓存层，MySQL 8.0 做主库
- ❌ MongoDB 方案否决：不适合强事务场景
```

**反例**（禁止 ❌）:
```markdown
## Redis 缓存方案 - 已确认
### 背景
我们讨论了三种方案...（省略200字过程描述）
### 方案对比
| 方案 | 优点 | 缺点 |
...（大段表格）
### 结论
最终选择了Redis...
```

### 长期记忆格式 (MEMORY.md)

```markdown
## [模块] - [决策/约定] (YYYY-MM-DD)
核心结论，一到两句话。（50-100 字）
```

**示例**（正确 ✅）:
```markdown
## 数据库选型 - MySQL + Redis (2026-03-12)
MySQL 8.0 主库 + Redis 7.0 缓存。事务一致性 + 高并发缓存需求。
```

**反例**（禁止 ❌）:
```markdown
## 数据库选型 - MySQL + Redis (2026-03-12)
### 核心方案: MySQL 8.0 作为主数据库...
### 关键架构认知: 数据库层分为...
### 用户确认决策: 用户在2026-03-12确认...
### 选型理由: 经过对比MongoDB、PostgreSQL...
```

### 容量控制

| 指标 | 上限 | 超限处理 |
|------|------|---------|
| 单条日志记录 | ≤ 3 行 | 压缩为结论 + 状态标记 |
| 单日日志文件 | ≤ 50 行 | 合并同主题条目 |
| MEMORY.md 单条目 | ≤ 100 字 | 精简为规则化表述 |
| MEMORY.md 总条目 | ≤ 30 条 | 使用 `/memory:tidy --compact` 整理 |

## 错误处理

### 日志文件不存在

先创建目录 `.codebuddy/memory/`（如不存在），再用 `write_to_file` 创建当日日志文件。

### MEMORY.md 不存在

用 `write_to_file` 创建 `.codebuddy/MEMORY.md`，写入基本结构后再追加内容。

### 追加内容失败

如果 `replace_in_file` 失败（如文件内容为空或匹配不到），改用 `write_to_file` 重写整个文件（先读取旧内容保留）。
