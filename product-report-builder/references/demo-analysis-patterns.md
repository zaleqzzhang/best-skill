# Demo 分析与文档生成范式

本文档是 Step 0（分析 Demo → 自动生成汇报文档）的核心参考，包含**信息提取规范、分节内容模板、句式范式库、质量自检清单**四大部分。

---

## Part 1: 信息提取清单

在生成任何文档内容之前，必须先从 Demo 中提取以下信息并填入结构化字段。

### 1.1 必填字段（无此项则无法生成）

| 字段 | 提取来源 | 示例 |
|------|---------|------|
| `产品名称` | Demo 标题 / 入口页 h1 / 目录名 | "群SVIP" / "AI试聊" |
| `产品类型` | 从功能推断 | "智能增值服务工具" / "AI关系破冰产品" |
| `目标用户` | Demo 中的用户角色 / 场景描述 | "QQ群主" / "想认识新朋友的QQ用户" |
| `核心定位一句话` | 综合提炼，≤20字 | "聚焦多群管理与内容运营的智能增值服务工具" |
| `Demo 页面列表` | 扫描目录所有 .html 文件 | index.html, batch-manage.html, ... |

### 1.2 推荐字段（有则大幅提升质量）

| 字段 | 提取方式 | 示例 |
|------|---------|------|
| `痛点列表` (3-6个) | 从 Demo 功能反推解决的问题 | 跨群管理难、单群管理累... |
| `功能模块列表` | 每个 Demo 页面对应一个功能 | 多群批量管理、自动托管... |
| `定价信息` | Demo 中如有价格/套餐页面 | ¥50/月, 5个群基础额度 |
| `竞品信息` | 如 Demo 有对比页面 | 里德助手、微友助手... |
| `团队名称` | 用户指定或默认值 | "QQ增值团队" |
| `设计原则/理念` | Demo 中如有关键词强调 | 隐私优先、零门槛... |

### 1.3 提取操作步骤

```
1. 读取 index.html → 提取产品名称 + 一句话定位
2. 列出所有 .html → 排除 index 后逐一快速浏览
   - 每个页面记录: 文件名 + 页面标题(h1) + 核心功能(1句话)
3. 反推痛点: 每个功能解决什么问题? → 去重合并为 3-6 个痛点
4. 检查是否有定价/竞品/设计原则相关页面
5. 将以上信息整理为结构化字段表
```

## Part 2: 分节内容生成模板

每节都有 **输入要求 → 输出格式 → 句式范式 → 禁止事项** 四层约束。

### 2.1 封面 (C1)

**输入**: 产品名称、定位词、类型、团队、日期

**输出格式**:
```html
<div class="cover">
  <div class="tag">{{cover_tag}}</div>
  <h1>{{产品名称}}产品规划</h1>
  <div class="subtitle">{{定位一句话}}</div>
  <div class="meta">{{年份}}年{{月份}} · {{团队名称}}</div>
</div>
```

**命名范式**:
- `{{cover_tag}}`: 固定 "产品规划汇报"
- `h1标题`: `{产品名称}产品规划`
- `subtitle`: 格式为 **"聚焦{核心领域}的{产品类型}"**
  - ✅ "聚焦多群管理与内容运营的智能增值服务工具"
  - ✅ "基于AI对话的关系破冰与社交匹配产品"
  - ❌ "一个很好的产品" （无实质信息）
  - ❌ 太长超过25字

**渐变色选择规则**:

| 产品调性 | 渐变方向 | 色值 |
|----------|---------|------|
| 效率工具/管理类 | 蓝→紫 | `#12B7F5 → #722ED1` |
| AI/科技/创新 | 紫→深紫 | `#667eea → #764ba2` |
| 社交/生活服务 | 青→蓝 | `#00B42A → #12B7F5` |
| 娱乐/游戏 | 橙→红 | `#FF7D00 → #F53F3F` |
| 金融/安全 | 深蓝→靛 | `#165DFF → #722ED1` |

> 不确定时默认使用蓝紫 (`#12B7F5 → #722ED1`)。

### 2.2 章节①: 产品定位 (C2 + C3 + 受众标签 + C5痛点 + 可选C9)

#### 2.2.1 定位描述块 (C3)

**输入**: 产品类型、目标用户、核心手段、解决的核心问题

**输出格式**:
```html
<div class="positioning">
  {产品名}是面向<strong>{目标用户}</strong>的<strong>{产品类型}</strong>，
  旨在通过<strong>{核心手段}</strong>，解决<strong>{核心问题}</strong>。
  {补充说明: 在XX的基础上，针对XX需求，提供XX能力。}
</div>
```

**句式范式 — 必须严格遵循此结构**:
```
第一段（定义）: {产品名}是面向<strong>{目标用户}</strong>的<strong>{产品类型}</strong>，
                旨在通过<strong>{核心手段}</strong>，解决<strong>{核心问题}</strong>。

第二段（边界）: 整个产品是在{基础平台/现有能力}的{基础功能}之上，
                专门围绕{进阶需求}，针对性设计的{差异化描述}。
                在不影响{基础体验}的前提下，为{目标用户}提供{价值总结}。
```

**字数限制**: 第一段 ≤ 50字，第二段 ≤ 80字

**示例对照**:
```
✅ 群SVIP是面向<strong>QQ群主</strong>的<strong>增值服务工具</strong>，
   旨在通过<strong>自动化与AI技术</strong>，解决<strong>群主在深度运营中的核心痛点</strong>。
   整个产品是在QQ群<strong>基础侧的群管理功能</strong>之上，专门围绕
   <strong>跨群管理、自动托管、内容生成</strong>等进阶群管理需求，
   针对性设计的增值服务。在不影响基础管理体验的前提下，
   为QQ群主提供更丰富、完善的管理服务。  (98字)

❌ 群SVIP是一个很好的帮群主管群的工具，有很多功能很实用。 (空泛，无结构)
```

#### 2.2.2 目标受众标签云 (`.audience-tags`)

**提取规则**: 从 Demo 的使用场景 / 用户故事 / 目标人群描述中提取

**格式**: 每个标签 = `{场景}+{群体}`，8-12字
```html
<div class="audience-tags">
  <span>明星/网红粉丝社群</span>
  <span>游戏/二次元兴趣群</span>
  <!-- 3-6个 -->
</div>
```

**禁忌**:
- ❌ 太宽泛: "所有人"、"QQ用户"
- ❌ 太窄: 仅1个标签（至少3个以体现覆盖面）
- ❌ 超过15字的标签

#### 2.2.3 痛点网格 (C5) — 三选一

**选项 A: 痛点四宫格** (≤4 个痛点，每个简短)
- 条件: 痛点数量 ≤ 4 且每个描述 ≤ 12 字
- 使用 emoji 图标 (😰 😫 🎯 🧩 💰 ⏰ 📊 🔗)
- 标题格式: **4字短语**（名词+形容词/动词）
- 描述: **10-15字**，说明具体困扰

```html
<div class="pain-point">
  <div class="icon">😰</div>
  <h4>跨群管理难</h4>        <!-- 4字 -->
  <p>同时管理多个社群，信息同步依赖手工逐群操作，跨群重复劳动繁琐</p>  <!-- ~22字 -->
</div>
```

**选项 B: 痛点两列网格** (>4 个痛点或描述较长)
- 使用 `.pp-num` 数字编号
- 标题可稍长（4-6字）
- 描述可达 20-30 字

**选项 C: 痛点卡片版** (AI试聊风格) 🆕
- 条件: 需要突出每个痛点的标题，且描述较长（20-40字）
- 使用 `.pp-header` + `.pp-num` 组合
- 每个痛点像一张小卡片，带编号徽章 + 大标题 + 详细描述
- 适合: 痛点需要详细解释、有因果逻辑的场景

```html
<div class="pain-points" style="grid-template-columns: repeat(2, 1fr);">
  <div class="pain-point">
    <div class="pp-header">
      <div class="pp-num">1</div>
      <h4>社交门槛高</h4>
    </div>
    <p>主动加好友后不知道说什么，<strong>破冰成功率低</strong>，导致社交关系难以建立。</p>
  </div>
  <div class="pain-point">
    <div class="pp-header">
      <div class="pp-num">2</div>
      <h4>话题难延续</h4>
    </div>
    <p>初次对话后话题容易冷场，<strong>缺乏持续互动的触发点</strong>，关系难以深入。</p>
  </div>
</div>
```

**三种样式选择指南**:

| 条件 | 推荐样式 | 理由 |
|------|---------|------|
| ≤4 个痛点 + 每个 ≤15 字 | 四宫格 emoji 版 | 简洁直观 |
| >4 个痛点 或 描述 ≤25 字 | 两列数字编号版 | 信息量适中 |
| 痛点需详细解释 + 描述 >25 字 | **卡片版** | 强调标题，支持长描述 |

**痛点撰写范式**:
```
标题: {痛点维度}{痛点性质}    ← 维度(跨群/单群/促活/内容...) + 性质(难/累/单一/难沉淀...)
描述: {具体表现}，导致{后果}
```

**禁止事项**:
- ❌ 用产品功能反向写痛点（"没有批量管理功能"——这是解决方案不是痛点）
- ❌ 痛点之间高度重叠
- ❌ 没有后果说明（只说现象不说不便之处）

#### 2.2.4 [可选] 设计原则/理念标签 (`.principle-tags`)

当产品有明确的设计哲学或核心理念时使用（从 Demo 的 about/design 页面提取）：
```html
<div class="principle-tags">
  <div class="principle-tag">
    <div class="pt-icon">{emoji}</div>
    <div class="pt-name">{原则名}</div>
    <div class="pt-desc">{1句解释}</div>
  </div>
  <!-- 3-5个 -->
</div>
```
- 名称 2-4 字（隐私优先 / 零门槛 / 自然生长...）
- 描述 ≤ 18 字

### 2.3 章节②: 产品方案 (C2 + 功能卡×N + C7亮点框)

#### 2.3.1 功能卡片 (`.feature-card`) ⭐ 最关键组件

**每个功能卡片的生成必须完成以下 4 步**:

**Step A: 确定功能名称和序号**
- 格式: `①{功能名}` / `②{功能名}` ...
- 名称风格: **动宾结构** 或 **名词短语**，4-6字
  - ✅ "多群批量管理"、"自动托管服务"、"一键生成群物料"
  - ❌ "功能1"、"管理"、"关于管理的功能"

**Step B: 选择图标 emoji**
- 从功能语义选择最贴切的 emoji（1个）
- 常用映射:
  - 管理/批量 → 📋 / 📂 / 🗂️
  - 自动/AI → 🤖 / ⚡ / 🧠
  - 内容/创作 → 🎨 / ✍️ / 📝
  - 活动/互动 → 🚀 / 🎯 / 🎪
  - 数据/分析 → 📊 / 📈 / 🔍
  - 设置/中心 → 👑 / ⚙️ / 🔐

**Step C: 填充三行内容**

| 行 | 字段 | 句式范式 | 字数 |
|----|------|---------|------|
| 核心机制 | `.label`="核心机制" | **"{手段/技术}，{实现方式}"** | 15-30字 |
| 主要功能 | `.label`="主要功能" | **"{功能1}、{功能2}、{功能3}"**，逗号分隔的功能列表 | 15-40字 |
| 业务价值 | `.label`="业务价值" | **"让用户{受益}，{量化效果}"** | 15-35字 |

**业务价值句式库**（必须从中选用或参照改写）:
```
降低 XX 成本（时间成本 / 人力成本 / 运营成本 / 制作成本）
提升 XX 效率（管理效率 / 运营效率 / 决策效率 / 匹配效率）
实现 XX 自动化（半自动化 / 全自动化 / 智能化）
解决 XX 问题（核心痛点 / 长期困扰 / 关键瓶颈）
覆盖 XX 场景（多场景 / 全流程 / 端到端）
将 XX 降至近乎为零（制作门槛 / 操作难度 / 学习成本）
大幅降低 XX（沟通成本 / 执行成本 / 试错成本）
```

**Step D: 绑定 data-demo**
- 选择最能展示该功能的 Demo 页面文件名
- 如果该功能涉及多个页面，绑定最主要的那一个

#### 2.3.2 [可选增强] 功能卡片内嵌组件

**`.fc-label` 小标签**: 当需要区分功能类型时添加
- 取值: "核心功能"、"入口载体"、"增值能力"
- 放在 `.feature-header` 内，`<h3>` 之前

**`.flow-step` 流程可视化**: 当功能涉及清晰的多步用户操作时
```html
<div class="flow-step">
  <div class="flow-item">{emoji} {步骤名1}</div>
  <div class="flow-arrow">→</div>
  <div class="flow-item">{emoji} {步骤名2}</div>
  <div class="flow-arrow">→</div>
  <div class="flow-item">{emoji} {步骤名3}</div>
</div>
```
- 放在"主要功能"行的 `<div class="value">` 内部
- 步骤数 3-5 个为宜

#### 2.3.3 核心亮点框 (C7, `.highlight-box`)

**放在所有功能卡片之后**，总结整个方案层级的差异化优势。

**句式范式**:
```
<h4>● 核心亮点</h4>
<ul>
  <li><strong>{差异化手段}</strong>，主打<strong>{1-2个高阶痛点}</strong>，
      和{竞品/基础能力}做出差异</li>
  <li><strong>{第二亮点}</strong>，{具体价值描述}</li>
</ul>
```

**规则**:
- 2-3 条，不超过 3 条
- 每条必须包含至少 1 个 `<strong>` 加粗关键词
- 必须回答 "凭什么这个方案能成"
- ❌ 不要写功能罗列（那是功能卡干的活），要写**洞察级结论**

**路线图式展示**:

<div class="highlight-box">

</div>

## Part 3: 全局撰写禁令

以下是在生成文档内容的任何位置都**绝对不允许**出现的内容：

### 3.1 禁用词汇表

| 禁止使用 | 替换为 | 原因 |
|----------|--------|------|
| 非常好 / 很好 / 很棒 | 具体优势描述 | 无信息量 |
| 非常重要 / 很重要 | "核心"/"关键"+理由 | 主观空洞 |
| 大幅提升 / 显著提升 | "提升 X%" 或具体描述 | 无法验证 |
| 用户痛点 / 用户需求（泛指） | 具体列出是什么痛/什么需 | 过于笼统 |
| 等等 / 等等 | 直接列举完或用"包括" | 不专业 |
| 我们认为 / 我觉得 | 基于数据/逻辑的判断 | 主观性 |
| 可能会 / 应该会 | "预计"/"预期"+条件 | 不确定性过高 |
| 不错的 / 还可以 | 具体评价维度 | 无标准 |
| 强大 / 完善 / 丰富 | 具体强在哪/完善在哪/丰富了什么 | 形容词堆砌 |

### 3.2 禁止模式

- ❌ **纯功能罗列**: 只写了"有什么功能"，没写"为什么要有"
- ❌ **自说自话**: 全篇没有提到目标用户的具体场景
- ❌ **数字缺失**: 该有数字的地方用了模糊词（"多个"、"大量"、"很快"）
- ❌ **段落过长**: 任何一段 > 4 行（含标点）
- ❌ **缺少加粗**: 连续 3 行正文没有任何 `<strong>` 关键词

## Part 4: 文档质量自检清单

Step 0 产出文档片段后、进入 Step 1 前，**必须逐条检查**。任一条不通过则返回修改。

### 4.1 结构完整性 (全部通过才能继续)

- [ ] 封面(C1)存在且包含 tag + h1 + subtitle + meta 四要素
- [ ] 至少有 2 个编号章节（① + ②）
- [ ] 每个章节都有 `.section-title`（圆形数字徽章 + h2）
- [ ] 页脚存在且有团队名 + 日期

### 4.2 封面质量

- [ ] subtitle ≤ 25 字且包含产品定位关键词
- [ ] 渐变色与产品调性匹配（非随意选取）
- [ ] meta 格式正确: "{YYYY}年{M}月 · {团队名}"

### 4.3 产品定位章节

- [ ] 定位描述块遵循两段式结构（定义 + 边界）
- [ ] 定位描述中 ≥ 3 处 `<strong>` 加粗
- [ ] 目标受众标签 ≥ 3 个且每个 ≤ 12 字
- [ ] 痛点网格选择了正确的布局（4列emoji vs 2列数字）
- [ ] 每个痛点标题为 4 字短语
- [ ] 痛点之间无明显重复

## Part 6: 数据可信度与引用规范 ⭐ 防编造核心规则

> **铁律：不知道的就不写。只写确定的信息。宁可少写，不可编造。**
>
> 汇报材料是给老板/同事看的专业文档，**绝不允许出现以下内容**：
> - ❌ `【待验证】`、`【待补充】`、`【来源待确认】`
> - ❌ `XX%`、`XX万` 这类占位符
> - ❌ `【以最终PRD为准】`、`【需用户提供】` 之类的免责声明
>
> 以上任何形式出现在汇报文档中都是**不专业的表现**。

### 6.1 三层处理策略

当遇到不确定的数据时，按以下优先级处理：

| 优先级 | 策略 | 做法 | 示例 |
|--------|------|------|------|
| **1（首选）** | **只写定性结论** | 用确定性措辞描述方向和趋势，**不含具体数字** | 不写"提升30%转化率" → 写"有效提升社交转化率"；不写"覆盖100万用户" → 写"覆盖广泛用户群体" |
| **2（次选）** | **直接删除该信息** | 如果定性结论也站不住脚，整段删除 | 删除整个未经验证的定价参数段落 |
| **3（禁止）** | **❌ 占位符/免责标注** | 永远不要用 | 禁止：`XX%`、`【待验证】`、`【以PRD为准】` |

### 6.2 可以写的 vs 不能写的

#### ✅ 可以写（有依据）

| 类型 | 条件 | 示例 |
|------|------|------|
| Demo 中可见的功能数量 | 数一下 Demo 页面里的实际元素 | "提供**3种**欢迎方式"、"支持**5+**个批量操作" |
| 用户明确告诉你的数据 | 用户在对话中给出的数字 | 用户说"目标50万DAU"，就写50万 |
| 产品本身的设定值 | Demo 或 PRD 里写死的参数 | "每次对话**10轮**"、"每天**3次**免费" |
| 行业公认的事实 | 无需证明的常识 | "社交产品普遍面临冷启动挑战" |
| 定性功能描述 | 描述产品**能做什么**（不是效果） | "AI根据双方资料生成破冰话题"、"支持一键发起试聊" |

#### ❌ 不能写（无依据）

| 类型 | 错误示例（❌） | 为什么错 |
|------|--------------|---------|
| 未经验证的效果数据 | "预计**提升30%**转化率" | 数字是 AI 编的 |
| 未验证的市场数据 | "破冰成功率**低于30%**" | 没有任何来源 |
| 未验证的用户特征 | "**Z世代**对此更敏感" | 泛化断言，无法证实 |
| 因果关系断言 | "因此**大幅提升**DAU" | 单功能不会自动带来结果 |
| 竞品判断 | "*竞品功能单一*" | 未做调研不能下结论 |
| 任何占位符 | "提升**XX%**"、"覆盖**XX**用户" | 汇报材料里出现占位符极其不专业 |
| 任何免责声明 | "【**以PRD为准**】"、"【**待验证**】" | 同上 |

### 6.3 定量 vs 定性的判断标准

**核心问题：这句话里的数字从哪来的？**

```
Demo 里能看到？    → ✅ 可以写定量
用户明确说了？     → ✅ 可以写定量
产品设定的参数？   → ✅ 可以写定量（如定价、次数限制）
以上都不是？       → ❌ 删掉数字，改为定性；如果定性也没把握 → 整句删除
```

**定性写法范例**：

| ❌ 编造定量（删掉数字） | ✅ 改为定性 |
|------------------------|-----------|
| 预计提升转化率**30%+** | 有效提升社交转化率 |
| 破冰成功率**低于30%** | 破冰成功率低 |
| 覆盖**100万+**用户 | 覆盖广泛用户群体 |
| 响应时间**<2秒** | 快速响应 |
| 日活达**50万** | （直接删除，不写） |

### 6.4 业务价值行的特殊处理

业务价值行是最容易出问题的位置（喜欢写"提升XX%""降低XX成本"）。

**安全写法 — 只用定性范式**：

- ✅ "有效解决 XX 问题"
- ✅ "显著提升 XX 效率" → 等等，"显著"在禁用词表里
- ✅ "提升 XX 转化率"（不加数字）
- ✅ "降低 XX 决策成本"
- ✅ "实现 XX 全流程自动化"
- ✅ "覆盖 XX 场景需求"

**判断公式**：业务价值 = **这个功能做了什么** + **带来什么方向的改变**（不量化程度）

### 6.5 定价数据的处理

定价信息只有两种情况可以写：

1. **Demo 里有明确的定价页面** → 按 Demo 内容如实抄写
2. **用户在对话中给出了定价方案** → 按用户给的写

**如果 Demo 里没有定价页面 + 用户没给定价信息 → 整个商业化章节不写**，不要编造"免费3次""SVIP10次"这种数字。

### 6.6 自检清单：数据红线（一票否决）

生成文档后，逐条检查。**任何一条不通过 = 文档不合格，必须修正后重新输出。**

- [ ] **全文不含任何形式的占位符**（`XX%`、`XX万`、`XX元`、`...` 等）
- [ ] **不含任何免责/待验证标注**（`【待验证】`、`【待补充】`、`【以PRD为准】`、`【需确认】`等）
- [ ] **每个百分比数字都能回答"这个数从哪来的"**（Demo? 用户? 产品设定?）
- [ ] **每个绝对数值都有明确来源**（同上）
- [ ] **所有定性结论都是保守表述**（不做无法支撑的强断言）
- [ ] **不存在"删掉数字后就变成空话"的句子**（如"显著提升"已被禁用，那整句需要重写而非只删词）
- [ ] **业务价值行全部使用安全的定性范式**（见 6.4）
- [ ] **定价/商业化章节只在有可靠来源时才出现**

## Part 6: 纯文档模式（无 Demo）特殊指南 🆕

当用户**没有 Demo HTML**，仅基于文字输入生成汇报页时使用本部分补充规则。

### 6.1 信息来源差异

| 字段 | 模式 A（有 Demo） | 模式 B（无 Demo） |
|------|------------------|-------------------|
| 产品名称 | 从 Demo 提取 | 用户口述 / PRD 标题 |
| 功能列表 | 扫描 Demo 页面 | **用户逐个描述** |
| 痛点推导 | 从功能反向推导 | **用户直接告知**或从场景推断 |
| 定价信息 | Demo 定价页面 | **用户提供**或跳过 |
| 视觉素材 | 无 | 无 |

### 6.2 功能卡片生成差异

模式 B 的功能卡与模式 A **组件相同，但有以下限制**：

| 属性 | 模式 A | 模式 B |
|------|--------|--------|
| `data-demo` | ✅ 必须绑定 | ❌ 不绑定 |
| `.click-hint` 元素 | ✅ 有 | ❌ 不加 |
| `cursor: pointer` | ✅ 可点击 | ❌ 默认 |
| 三行内容（机制/功能/价值） | ✅ 相同 | ✅ 相同 |

### 6.3 痛点网格布局建议

模式 B 下默认使用 **2 列布局**（阅读体验优于4列）：
- `{{pain_columns}}` 设为 `2`

- 每行描述可以更长（纯文档不需要考虑手机预览区的宽度限制）

### 6.4 纯文档模式的典型触发话术

用户说以下任何一句话时，进入模式 B：

- "没有 Demo"、"来不及做 Demo"、"还没开始做页面"

- "只有 PRD/需求文档"
- "先出文字版，后面再补 Demo"

## Part 7: 实战踩坑经验库 🆕

> 本区域由 AI 在实际使用中自动积累。当遇到**需要 2 次及以上尝试才解决的问题**时，应简要记录经验，方便后续复用。

### 7.1 跨域 / 协议兼容

- **postMessage origin 问题**: 在 `file://` 协议下，`window.location.origin` 返回字符串 `'null'`，但实际 origin 是 `null`，导致 `postMessage` 失败。**解决方案**：必须使用 `'*'` 作为 targetOrigin，不能用 `window.location.origin`。
- **iframe contentWindow 不可读**: 本地 `file://` 打开时，跨域 iframe 的 `contentWindow.location` 不可读，需要通过 `postMessage` 间接通信。

### 7.2 CSS 兼容

- **backdrop-filter**: `backdrop-filter: blur()` 在某些浏览器需要 `-webkit-` 前缀，建议同时写两行：
  ```css
  -webkit-backdrop-filter: blur(4px);
  backdrop-filter: blur(4px);
  ```
- **:has() 选择器**: `:has()` 选择器在旧版浏览器不支持，如需兼容，改用 JS 添加类名或使用父元素类控制。

### 7.3 移动端适配

- **iPhone 安全区域**: 如果页面需要在 iOS 设备上全屏显示，需要处理 `safe-area-inset-*` 避免内容被刘海/底部条遮挡。
- **触摸事件**: 移动端 hover 效果不明显，功能卡片的 `.click-hint` 在移动端可考虑默认显示。

### 7.4 布局问题

- **flex 溢出**: 左右双栏布局中，如果左侧内容过长导致溢出，需要在 `.doc-panel` 上设置 `min-width: 0` 阻止 flex 子项撑开。
- **iframe 高度**: iPhone 外壳内的 iframe 使用 `height: 100%` 需要确保父容器有明确高度，否则可能显示为 0。

### 7.5 交互绑定

- **data-demo 匹配**: `data-demo` 属性值必须是**纯文件名**（如 `page8.html`），不能带路径前缀（如 `./page8.html`）。
- **导航同步**: 右侧导航 Tab 的 `data-page` 和 `.feature-card[data-demo]` 要使用完全相同的文件名格式。
