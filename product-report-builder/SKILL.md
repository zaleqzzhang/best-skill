# Product Report Builder

## Overview

本 Skill 支持两种模式：

### 模式 A：Demo + 文档 双栏展示（默认）
从 **产品 Demo → 汇报文档 → 左右双栏页面**：
- 左侧：结构化产品文案（封面→定位→痛点→功能→商业化）
- 右侧：iPhone 外壳内嵌 Demo 实时预览，支持点击跳转

### 模式 B：纯文字汇报页（无 Demo）
当用户**来不及做 Demo 或没有 Demo** 时，仅基于文字输入生成全宽居中的精美文档页面：
- 全宽布局，无需 HTTP 服务，双击即可打开
- 复用全部 14 种组件样式，功能卡不可点击（无 data-demo 绑定）
- 可后续补充 Demo 升级为模式 A

**内容质量保障**（两种模式通用）:
- 文档生成遵循**结构化句式范式**（定位描述两段式、业务价值标准句式库、痛点撰写模板）
- 执行**一票否决制质量自检**（禁用词汇、加粗密度、段落长度、data-demo 完整性）
- **数据可信度强制校验**（禁止编造量化指标/定性结论，所有数字必须标注来源，未验证数据用占位符标记）
- 参考**两套真实案例**的组件选择和风格差异（群SVIP / AI试聊）

输出文件可直接部署至 HTTP 服务器访问，用于向上级/同事宣讲介绍。

## Design Specifications

| 规范项 | 模式 A（双栏） | 模式 B（纯文档） |
|--------|---------------|-----------------|
| 布局 | 左右双栏 | **全宽居中，单栏** |
| 左侧/内容宽度 | `flex: 1` / 最大 800px | **最大 860px 居中** |
| 右侧预览区 | 480px + iPhone 外壳 | **无** |
| 功能卡交互 | 可点击跳转 Demo | **不可点击** |
| data-demo | 必须绑定 | **不绑定** |
| HTTP 服务 | **可选**（双击即可打开） | **不需要**（双击即用） |
| 背景色 | 左 `#F5F7FA` / 右深色 `#1D2129` | **纯 `#F5F7FA`** |
| 主色 | `#12B7F5`，iOS 风格 | 同左 |
| 团队署名 | 默认 "QQ增值团队" | 同左 |

## Workflow (7 步)

### 模式判断（在 Step 0 之前执行）

**首先确认用户是否有 Demo**：

| 问题 | 回答 |
|------|------|
| 用户提供了 Demo HTML 文件/目录？ | → **模式 A**：走完整流程（Step 0-6），使用 `report-template.html` |
| 用户只有文字描述 / PRD / 口述要点？ | → **模式 B**：跳过 Demo 分析，走纯文档流程（见下文） |

### 【模式 A】Step 0: 分析 Demo 自动生成文档 ⭐ 核心步骤

**触发条件**: 用户提供了 Demo HTML 但未提供现成文档内容。

本步骤是整个 Skill 的**内容质量命脉**。必须严格按照 `references/demo-analysis-patterns.md` 中的范式执行。

#### 0.1 信息提取（结构化字段填充）

**必须**先完成 `Part 1: 信息提取清单` 中所有必填字段的提取：
- 从 index.html 提取产品名称、定位词
- 扫描目录获取所有 .html 文件列表及每个页面的功能概述
- 反推痛点列表（3-6 个，从功能反向推导解决什么问题）
- 检查定价/竞品/设计原则等可选信息

> 将提取结果整理为结构化字段表后再进入下一步。不要跳过此步直接写文案。

#### 0.2 分节内容生成

按照 `Part 2: 分节内容生成模板` 逐节生成，每节必须遵守：

| 章节 | 必用组件 | 关键约束 |
|------|---------|---------|
| **封面 C1** | `.cover` | subtitle ≤25字；渐变色按产品调性选 |
| **① 产品定位** | C2编号标题 + **C3定位描述块(两段式)** + 受众标签 + C5痛点 | 定位描述严格遵循"定义+边界"句式模板 |
| **② 产品方案** | C2编号标题 + **功能卡×N(C6)** + C7亮点框 | 每张卡必须有 data-demo + 三行齐全 + 业务价值使用句式库 |
| **③ 商业化/计划** | C2编号标题 + 定价组件 或 路线图 | 有定价则含对比卡+说明；无定价则写里程碑 |
| **页脚** | `.footer` | 团队名+日期 |

**每个功能卡片生成时必须完成 4 步**: 确定名称序号 → 选 emoji → 填充三行(机制/功能/价值) → 绑定 data-demo

#### 0.2.5 数据可信度校验 ⭐ 防编造关键步骤

分节生成完成后、进入自检前，**必须**执行 `Part 6: 数据可信度与引用规范`：

1. **提取所有数字** — 找出文中每一个百分比/金额/时长/数量/比率
2. **逐个核查来源** — 每个数字必须能回答"从哪来的"
   - Demo 可见 / 用户明确给出 / 产品设定参数 → ✅ 保留
   - 以上都不是 → ❌ **删掉数字，改为定性结论；如果定性也没把握，整句删除**
3. **扫描定性结论** — 找出所有因果句和效果断言，无依据的改写为纯功能描述
4. **红线扫描** — 检查是否残留任何占位符（`XX%`）、免责标注（`【待验证】`），有则立即删除

> **铁律：不知道的就不写。只写确定的信息。宁可少写，不可编造。汇报材料里绝不允许出现「待验证」「XX%」之类的内容。**

#### 0.3 全文质量自检（一票否决）

生成完成后，**必须逐条执行 `Part 4: 文档质量自检清单`**：

**一票否决项**（任一不通过则返回修改）:
- 全文含禁用词汇（非常好/很重要/强大/完善...）
- 连续 3 行正文无 `<strong>`
- 任何一段 > 4 行
- 功能卡缺少 `data-demo` 属性
- 痛点只写现象没写后果
- 业务价值未使用标准句式库
- **存在无来源的具体数字（百分比/金额/数量/比率）— 见 Part 6 数据校验规则**
- **存在编造的定性结论或因果断言（"因此提升XX"/"显著改善XX"等）**

**完整性检查**:
- 结构完整性 7 项全部通过
- 封面质量 3 项全部通过
- 各章节专项检查全部通过
- 交互绑定(data-demo 匹配)验证通过

#### 0.4 产出文档 HTML 片段

将自检通过的文档内容保存为 HTML 片段（封面到页脚），供 Step 3 注入 `{{document_content}}`。

### Step 1: 确认输入材料

如果 Step 0 已自动生成了文档，此步简化为确认：

1. **Demo HTTP URL** — 确保可通过 HTTP 访问
   - 本地路径需启动 HTTP 服务器
   - 注意：file:// 会跨域拦截

2. **文档内容** — 使用 Step 0 生成的结果，或用户提供的已有文档

3. **右侧导航 Tab 列表** — 从 Demo 的 HTML 文件列表中提取，格式如：
   ```html
   <a href="#" data-page="index.html">🏠 首页</a>
   <a href="#" data-page="page2.html">📋 功能页</a>
   ```

### Step 2: 读取模板

读取 `assets/report-template.html` 作为基础骨架。

### Step 3: 替换占位符生成最终 HTML

替换以下占位符：

| 占位符 | 替换为 |
|--------|--------|
| `{{title}}` | 产品名称 |
| `{{cover_gradient}}` | 封面渐变色（默认蓝紫，可根据产品调性调整） |
| `{{cover_tag}}` | 封面 tag（默认"产品规划汇报"） |
| `{{cover_title}}` | 封面大标题 |
| `{{cover_subtitle}}` | 封面副标题（一句话概括产品定位） |
| `{{cover_meta}}` | 封面 meta 信息（日期+团队） |
| `{{document_content}}` | Step 0 生成的完整文档 HTML（封面到页脚之间的所有内容） |
| `{{demo_url}}` | Demo 入口页面**相对路径**（如 `page15.html`），**禁止使用 localhost 或绝对 URL**
| `{{preview_nav_items}}` | 右侧导航 Tab 的 `<a>` 标签列表 |
| `{{team_name}}` | 团队名称（默认"QQ增值团队"） |
| `{{primary_color}}` | 主色调（默认 #12B7F5） |

写入输出文件到工作目录。

### Step 4: 填充右侧导航与交互绑定

确保以下 JS 逻辑正确：

1. 导航 Tab 的 `data-page` 与实际 Demo 文件名匹配
2. 功能卡片 `.feature-card[data-demo]` 与 Demo 页面对应
3. Toast 提示正常工作
4. iframe 地址使用**相对路径**（DEMO_BASE = 空字符串），确保：
   - `{{demo_url}}` 填入相对文件名（如 `page15.html`）
   - JS 中 `DEMO_BASE` 设为空字符串
   - **禁止使用 localhost 或任何 HTTP URL**（保证 zip 发给别人也能直接打开）

#### ⚠️ file:// 协议兼容性（重要）

如果 Demo 页面内部有使用 `postMessage` 进行父子页面通信，**必须**将 targetOrigin 设为 `'*'` 而非 `window.location.origin`：

```javascript
// ❌ 错误写法（file:// 协议下会报错）
window.parent.postMessage({ action: 'switchPage', url: url }, window.location.origin);

// ✅ 正确写法（兼容 file:// 协议）
window.parent.postMessage({ action: 'switchPage', url: url }, '*');
```

**原因**：在 `file://` 协议下，`window.location.origin` 返回字符串 `'null'`，但浏览器实际 origin 是 `null`（非字符串），导致安全检查失败。使用 `'*'` 可以绕过此问题，对于本地文件是安全的。

### Step 5: 验证与预览

#### 快捷验证（推荐）
如果汇报 HTML 和 Demo 在同一目录，**双击 HTML 文件即可预览**，无需启动 HTTP 服务器。

#### 完整验证（如需调试 iframe 跨域问题）
1. 启动 HTTP 服务器（如 `python -m http.server 8080`）
2. 使用 preview_url 工具打开生成的 HTML

#### 检查清单
- [ ] 左右布局正常，无溢出
- [ ] 右侧 iframe 正确加载 Demo（如报跨域错误，检查 Demo 目录是否正确）
- [ ] 功能卡片 hover 显示"▶ 在右侧预览"
- [ ] 点击功能卡片跳转正确 Demo 页面
- [ ] 右侧导航 Tab 点击切换高亮
- [ ] 移动端响应式可用（≤1024px 切换上下布局）

### Step 6: 输出交付

**输出位置规则**：汇报 HTML 文件必须写入 **Demo 同目录**下，文件名格式为 `{产品名}产品规划汇报.html`。

例如 Demo 目录为 `C:\xxx\AI试聊demo\`，则输出：
```
C:\xxx\AI试聊demo\AI试聊产品规划汇报.html
```

完成后提供以下信息：
- 生成的 HTML 文件绝对路径
- **双击即可打开**（无需 HTTP 服务器，只要汇报页和 Demo 在同一目录）
- 可选：如需部署到远程服务器，打包整个 Demo 目录上传即可

### 【模式 B】纯文字汇报流程（无 Demo）

当用户**没有 Demo HTML**，仅通过文字描述/PRD/口述要点进行汇报时使用。

#### B.1 信息收集

收集以下信息（必填项用 * 标注）：

| 字段 | 必填 | 示例 | 说明 |
|------|:----:|------|------|
| **产品名称\*** | ✅ | "AI试聊产品规划" | 用于标题和文件名 |
| **一句话定位\*** | ✅ | "AI社交破冰工具——降低陌生人社交心理门槛" | 封面副标题 |
| 目标用户 | ✅ | "QQ陌生人社交场景下的年轻用户" | 定位描述块 |
| 核心痛点（3-6个） | ✅ | 见痛点模板 | 每个含4字标题+描述 |
| 功能模块列表 | ✅ | ["AI试聊引擎", "扩列卡系统"] | 决定功能卡片数量 |
| 每个功能的详情 | ✅ | 核心机制 / 主要功能 / 业务价值 | 填入功能卡三行 |
| 商业化模式 | | 定价方案 / 权益对比 / 转化逻辑 | 可选 |
| 竞品参考 | | 名称 + 类型标签 + 差异点 | 可选 |
| 团队名 | | 默认"QQ增值团队" | |
| 渐变色偏好 | | 蓝紫/紫深紫/绿青/其他 | |

> 如果用户信息不完整，用已知信息先生成，缺失部分留空或标注「待补充」。

#### B.2 分节生成（复用 Part 2 范式）

与模式 A 相同的章节结构：

1. **封面 C1** — 产品名 + 副标题 + meta
2. **① 产品定位** — C3 定位描述块(两段式) + C5 痛点网格 + 设计原则标签
3. **② 产品方案** — C6 功能卡 × N（**无 data-demo 绑定**）
4. **③ 商业化/计划** — 定价组件 或 里程碑路线图
5. **页脚**

**与模式 A 的关键差异**：
- 功能卡片：**不加** `data-demo` 属性、**不加** `click-hint` 元素、cursor 改为默认
- 不需要右侧预览区、不需要导航 Tab、不需要 iframe JS
- 痛点网格默认 **2 列**（纯文档阅读体验更好）

#### B.3 选择模板 & 替换占位符

1. 使用模板：**`assets/report-template-doc-only.html`**
2. 替换占位符（与模式 A 相同的字段）
3. **特殊处理**：
   - `{{pain_columns}}` → 设为 `2`
   - 功能卡片中**移除所有 data-demo 属性**
   - `{{preview_nav}}` → 留空（纯文档无此区域）

#### B.4 输出交付

**输出位置规则**：写入用户指定目录或当前工作目录，文件名 `{产品名}产品规划汇报.html`。

完成后提供以下信息：
- 生成的 HTML 文件绝对路径
- **双击即可打开**，无需 HTTP 服务
- 如后续补充了 Demo，可升级为模式 A 双栏版本

### C1. 封面 (`.cover`)

封面包含 tag + h1 + subtitle + meta 四要素。渐变背景通过 CSS 变量控制。

**推荐渐变组合**：蓝紫 `(#12B7F5→#722ED1)` / 紫深紫 `(#667eea→#764ba2)` / 绿青 `(#00B42A→#12B7F5)`

> 完整 HTML 结构见 `assets/report-template.html` 的 `.cover` 部分

### C2. 编号章节标题 (`.section-title`)

圆形数字徽章自动着主色，每个 section 需要 id 用于锚点。

> 完整 HTML 结构见 `assets/report-template.html` 的 `.section-title` 部分

### C2.1 子章节标题 (`.sub-section-title`) 🆕

当一个章节内需要分隔多个子模块时使用（如"核心痛点"与"核心策略"）。

```html
<div class="sub-section-title">🔴 核心痛点</div>
<!-- 痛点内容 -->

<div class="sub-section-title">✨ 核心策略</div>
<!-- 策略内容 -->
```

特点：
- 左侧带主色边框，醒目区分
- 字号 18px，略小于章节大标题
- 可选 emoji 前缀增强可读性

常用场景：
- 在"产品定位"章节内分隔痛点与策略
- 在"产品方案"章节内按功能分组
- 任何需要二级结构的地方

### C3. 定位描述块 (`.positioning`)

浅色渐变背景 + 彩色左边框，一段话概括（2-4行）。

> 详细句式范式见 `references/demo-analysis-patterns.md` Part 2.2

### C3.1 策略引导语 (`.strategy-intro`) 🆕

当章节（如"产品方案"）开头需要一段提纲挈领的引导语时使用。

```html
<div class="strategy-intro">
  针对上述痛点，我们设计了<strong>三大核心模块</strong>：
  AI引擎负责对话生成，扩列卡系统负责场景入口，匹配算法负责精准推荐。
</div>
```

特点：
- 橙色虚线下边框，与后续内容有视觉分隔
- 支持 `<strong>` 加粗关键词（显示为橙色）
- 1-2 句话即可，≤50 字

适用场景：
- 在功能卡片之前，解释整体设计思路
- 在痛点网格之前，说明为什么这些是核心痛点
- 任何需要"总—分"结构的地方

### C5. 痛点网格

**四宫格版 (emoji)**: 适用于 ≤4 个短痛点，`grid-template-columns: repeat(4, 1fr)`

**两列版 (数字编号)**: 适用于 >4 个或描述较长，`grid-template-columns: repeat(2, 1fr)`

> 痛点撰写范式见 `references/demo-analysis-patterns.md` Part 2.2.3

### 功能卡片 (`.feature-card`) ⭐ 核心组件

功能卡片是最关键的组件，每张卡片必须包含：
- `data-demo` 属性绑定右侧跳转
- 三行内容：核心机制 + 主要功能 + 业务价值
- 彩色左边框自动循环（蓝→绿→橙→紫）

**可选增强**：`.fc-label` 小标签、`.flow-step` 流程可视化

> 详细生成步骤见 `references/demo-analysis-patterns.md` Part 2.3.1

### 流程步骤可视化 (`.flow-step`) — 可选增强

放在 `.feature-row .value` 内部使用，适用于多步骤操作流程，步骤数 3-5 个为宜。

### C7. 核心亮点框 (`.highlight-box`)

黄色渐变背景，橙色强调。2-3 条要点，每条带加粗关键词。

> 详细句式范式见 `references/demo-analysis-patterns.md` Part 2.3.3

### 用户类型对比卡 (`.user-types` 或 `.pricing-cards`)

- **3列版** (`.user-types`): 群SVIP风格
- **2列版** (`.pricing-cards`): AI试聊风格
- featured 卡片自动显示"推荐"角标，价格用大号字体突出

### C14. 右侧导航 Tab (`#preview-nav`)

从 Demo 目录的 html 文件列表生成，emoji + 名称格式，第一个默认 active。
