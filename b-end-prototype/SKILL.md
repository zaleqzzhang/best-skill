# B-End Prototype — 中后台原型还原与设计技能

## 概述

此技能指导 AI 从用户提供的参考图中自动学习设计规范，将提取的设计令牌按产品隔离持久化存储，并基于学到的规范进行像素级原型还原，产出高保真交互原型（单个 HTML 文件）。已学习的设计规范跨会话持久保留，后续会话自动复用，无需重复提取。

**核心原则**：
1. **参考图即规范**：所有设计决策（色彩、字号、间距、圆角、阴影等）均从参考图中提取，不使用任何预设设计系统
2. **先提取后生成**：必须先完成设计规范提取，再开始编写代码
3. **交互内嵌**：多种状态（空态、loading、错误态等）通过 JavaScript 交互切换体验，**禁止**在页面上平铺罗列
4. **多状态实例可见**：当参考图中同一目标物存在多种状态变体（如"待审核/已通过/已驳回"），列表或容器中必须为每种状态至少提供一个实例，确保开发人员一目了然地看到各状态的视觉差异
5. **多产品持久化**：支持同时管理多个产品的设计令牌，按产品隔离存储，通过注册表索引和切换

## 多产品令牌架构

### 文件结构

```
SKILL.md                               ← 技能主文件，定义完整工作流程和设计规范
references/
└── ui_patterns.md                     ← 通用 UI 设计模式库（布局和交互模式知识，与设计系统无关）
assets/
├── token_schema.json                  ← 单产品令牌的 JSON Schema（约束令牌文件格式）
├── products_registry.json             ← 产品注册表（轻量索引，记录所有已学习产品）
└── tokens/                            ← 各产品的设计令牌文件夹
    ├── product_slug_a.json            ← 产品 A 的设计令牌
    ├── product_slug_b.json            ← 产品 B 的设计令牌
    └── ...
```

### products_registry.json 结构

```json
{
  "active_product": "product_slug_a",
  "products": {
    "product_slug_a": {
      "display_name": "XX管理后台",
      "file": "tokens/product_slug_a.json",
      "source_count": 5,
      "last_updated": "2026-03-27T17:00:00+08:00"
    },
    "product_slug_b": {
      "display_name": "YY运营平台",
      "file": "tokens/product_slug_b.json",
      "source_count": 3,
      "last_updated": "2026-03-26T10:00:00+08:00"
    }
  }
}
```

### 单产品令牌文件（tokens/xxx.json）

每个产品的令牌文件格式遵循 `token_schema.json`，与之前的 `learned_tokens.json` 结构完全一致：

```json
{
  "meta": {
    "product_name": "XX管理后台",
    "last_updated": "2026-03-27T17:00:00+08:00",
    "source_count": 5,
    "notes": "基于5张参考图提取"
  },
  "colors": { ... },
  "typography": { ... },
  "spacing": { ... },
  "border_radius": { ... },
  "shadows": { ... },
  "layout": { ... }
}
```

### 产品 slug 命名规则

- 由 AI 根据产品名称自动生成英文 slug
- 规则：小写字母 + 下划线，如 `xx_admin`、`yy_ops_platform`
- `display_name` 保留用户提供的中文原名

## 核心工作流程（MANDATORY）

### 第零步：初始化 — 读取注册表并确定目标产品（MANDATORY — 每次会话开始时执行）

**在做任何事情之前**，先读取 `{SKILL_DIR}/assets/products_registry.json`：

#### 场景 A：注册表为空（全新用户）

```
products 为空 → 没有任何已学习的产品
```

- 如果用户**提供了参考图**：询问产品名称（或从参考图/对话上下文推断），创建新产品，进入第一步
- 如果用户**未提供参考图**：进入"无参考图时的处理"流程

#### 场景 B：注册表中只有 1 个产品

- 自动选中该产品作为当前激活产品
- 告诉用户："已检测到 **{display_name}** 的设计规范（基于 {source_count} 张参考图），将自动使用。"
- 如果用户提供了新参考图：进入第一步，更新该产品的令牌
- 如果用户未提供参考图：直接使用已有令牌，跳到第三步

#### 场景 C：注册表中有多个产品

- **如果用户在对话中明确指定了产品名**（如"做XX后台的列表页"）：
  - 自动匹配并切换 `active_product`
  - 告诉用户："已切换到 **{display_name}**，使用其设计规范。"

- **如果用户未指定产品名**：
  - 列出所有可用产品，让用户选择：
    ```
    检测到您有多个产品的设计规范：
    1. XX管理后台（5 张参考图，最后更新 03-27）
    2. YY运营平台（3 张参考图，最后更新 03-26）
    
    当前激活的是：XX管理后台
    请问本次使用哪个产品的规范？（输入名称或序号，或直接开始默认使用当前激活的产品）
    ```

- **如果用户提供了参考图且产品名是全新的**：创建新产品，进入第一步

#### 场景 D：用户主动管理产品

用户可以使用以下指令管理产品：

| 用户说 | AI 做 |
|-------|-------|
| "切换到 YY 运营平台" | 更新 `active_product` → 确认切换 |
| "我有哪些产品的设计规范？" | 读取注册表 → 列出所有产品名 + 参考图数量 + 最后更新时间 |
| "删除 YY 运营平台的设计规范" | 删除对应 tokens/xxx.json + 从注册表中移除该产品 + 如果删的是 active_product 则切换到另一个或置空 |
| "重置所有设计规范" | 清空 tokens/ 文件夹 + 重置注册表 |
| "重命名 XX管理后台 为 XX后台系统" | 更新注册表中的 display_name + 更新令牌文件中的 product_name |

### 第一步：像素级分析参考图

若用户提供了参考截图，**逐像素**分析以下要素（这一步决定了最终还原度）：

1. **页面底色**：判断页面的实际底色（白色 #FFFFFF？浅灰 #F3F3F3？还是其他？），不要预设
2. **布局结构**：是否有顶部导航？左侧菜单？内容区域如何布局？是通栏还是居中容器？具体宽度/间距
3. **组件形态**：每个组件的精确样式——圆角大小、边框粗细及颜色、阴影层级、内外间距
4. **色彩运用**：从参考图中提取实际使用的所有颜色值，包括背景色、文字色、边框色、标签色、品牌色、功能色
5. **文字层级**：标题/正文/辅助文字的具体字号、字重、颜色、行高
6. **元素间距**：元素之间的精确间距，不要使用通用规范值覆盖参考图中的实际间距
7. **交互线索**：参考图中暗示的交互（如复选框暗示可批量操作、搜索框暗示过滤功能、Tab 暗示切换）

### 第二步续：持久化写入令牌（MANDATORY — 每次提取后执行）

完成第二步的设计规范提取后，**必须将结果写入对应产品的持久化文件**：

1. **确定目标产品**：
   - 如果注册表中已有该产品：读取对应的 `tokens/xxx.json`
   - 如果是新产品：生成 slug，创建 `tokens/{slug}.json`

2. **合并更新令牌文件**：
   - 如果是全新产品（文件刚创建）：直接写入所有提取的令牌
   - 如果已有令牌：逐字段对比，**仅更新有差异的部分**，保留已有令牌中本次参考图未涉及的字段

3. **更新令牌文件的元信息**：
   - `last_updated` → 当前时间
   - `source_count` → 累加 1
   - `product_name` → 保留已有的或使用用户告知的名称

4. **更新注册表**：
   - 新产品：添加到 `products` 字典
   - 已有产品：更新 `source_count` 和 `last_updated`
   - 将 `active_product` 设为当前产品的 slug

5. **写入文件**：
   - 令牌文件写入 `{SKILL_DIR}/assets/tokens/{slug}.json`，格式遵循 `{SKILL_DIR}/assets/token_schema.json`
   - 注册表写入 `{SKILL_DIR}/assets/products_registry.json`

6. **输出变更摘要**给用户：
   - 全新产品时：输出"已为 **{display_name}** 创建设计规范（保存为 tokens/{slug}.json），已保存 N 项设计令牌"
   - 更新已有产品时：输出"已更新 **{display_name}** 的设计规范，本次更新了以下令牌：[变更列表]"

### 第三步：确定页面类型并精确还原

根据参考图判断页面类型（可参考 `{SKILL_DIR}/references/ui_patterns.md`），使用对应的设计模式：

| 页面类型 | 核心组件 |
|---------|---------|
| **列表页** | 操作栏、筛选区、数据表格/卡片列表、分页 |
| **表单页** | 卡片式分组、标签+输入区、底部操作栏 |
| **详情页** | 基本信息卡片、Tab 页签、关联数据 |
| **仪表盘** | 数据卡片、图表组件、时间筛选 |
| **配置页** | 配置项分组、开关/选择器、保存操作 |

### 第四步：交互增强设计

在忠实还原参考图的基础上，通过 **JavaScript 交互** 增强以下体验。所有状态通过交互触发，**不在页面上平铺展示**：

**页面级状态切换**（通过页面右下角的「状态预览」浮动切换器让开发人员切换查看）：
- 正常态（默认展示，即参考图的还原）
- 空态：暂无数据时的引导展示，通过切换器切换查看
- 加载态：骨架屏 / Loading 动画，通过切换器切换查看
- 错误态：加载失败 + 重试按钮，通过切换器切换查看

**组件级交互**（通过 CSS 伪类和 JS 事件自然触发）：
- 按钮 hover/active 状态变化（悬停色从品牌色推算加深 10-15%）
- 输入框 focus 高亮（使用提取的品牌色作为焦点边框色）
- 表格行 / 卡片 hover 效果
- 弹窗/抽屉通过点击触发打开/关闭
- 删除等危险操作弹出确认弹窗（使用提取的错误状态色）
- 操作完成后显示 Toast 消息提示（使用提取的成功状态色）
- Tab/导航切换交互
- 搜索/过滤实时交互
- 复选框勾选 + 批量操作条

### 第五步：生成高保真原型

产出 **单个 HTML 文件**，内嵌 CSS 和 JavaScript，可直接在浏览器中打开。

**视觉还原要求**：

1. **严格使用第二步提取的设计令牌**：
   - 所有色值、字号、间距、圆角、阴影均来自第二步的提取结果
   - 禁止引入任何提取结果之外的"规范值"
   - 如遇第二步未覆盖的场景，从已提取的令牌中推算（如次要按钮色从品牌色推算浅色变体）

2. **组件精确度**：
   - Tab/导航组件：精确还原形状（胶囊/下划线/方块）、选中态样式、图标位置
   - 卡片：精确还原圆角、内边距、信息排列方式
   - 标签（Tag）：精确还原背景色、文字色、圆角、间距
   - 按钮：精确还原尺寸、颜色、位置

3. **布局精确度**：
   - 容器宽度/最大宽度与参考图一致
   - 网格列数/间距与参考图一致
   - 元素的对齐方式（左对齐/居中/右对齐/两端对齐）与参考图一致

## 无参考图时的处理

当用户未提供参考图时：

1. **优先检查注册表**：读取 `{SKILL_DIR}/assets/products_registry.json`：
   - 如果有 `active_product` 且对应令牌文件存在：直接使用该产品的令牌，告知"正在使用 **{display_name}** 的设计规范"
   - 如果有多个产品但未指定：列出产品列表让用户选择
2. 如果没有任何已学习的产品，**不要使用任何预设设计系统**，而是：
   - 询问用户是否有参考产品/网站可以截图提供
   - 如果用户确认没有参考，请用户提供以下最低限度的设计偏好：
     - 品牌色（一个主色即可）
     - 风格偏好（简约/圆润/紧凑）
     - 页面底色偏好（白底/灰底）
   - 基于用户提供的偏好信息生成设计令牌，然后按照正常流程生成原型
   - **并将用户偏好生成的令牌也按多产品架构写入持久化文件**（需要产品名）

## 状态切换器设计规范

在页面右下角添加一个固定定位的 **浮动工具条**：

```html
<!-- 状态切换器 - 供开发人员查看不同状态 -->
<div class="dev-state-switcher">
  <span class="dev-label">状态预览</span>
  <button class="dev-btn active" onclick="switchState('normal')">正常</button>
  <button class="dev-btn" onclick="switchState('empty')">空态</button>
  <button class="dev-btn" onclick="switchState('loading')">加载中</button>
  <button class="dev-btn" onclick="switchState('error')">错误</button>
</div>
```

样式规范：
- 固定在页面右下角（`position: fixed; bottom: 24px; right: 24px;`）
- 背景色 `#1A1A1A`，文字 `#FFFFFF`，圆角 8px
- 带有「状态预览」标签，避免与原型内容混淆
- 按钮激活态使用从参考图提取的品牌色
- z-index 高于所有弹窗层（如 2000）

点击按钮时，**同一内容区域**切换展示对应状态的内容（使用 `display: none/block` 或 `visibility` 控制），默认显示正常态。

## 页面类型还原指南

### 列表页

从参考图中学习以下要素：

- **操作栏**：从参考图提取按钮类型（实心/描边/文字）、颜色、排列方式（左侧主操作 vs 右侧工具），不预设固定色值
- **筛选区**：观察参考图中是否有筛选标签、搜索框、下拉筛选，还原其位置和交互形式
- **数据区**：判断参考图使用的是表格、卡片列表还是其他展示形式，还原其列定义、行高、对齐方式、悬停效果
- **分页**：还原参考图中分页组件的位置、样式和信息展示（总条数、页码、每页条数等）
- **批量操作**：如参考图有复选框，还原批量操作条的样式和出现方式

### 表单页

从参考图中学习以下要素：

- **布局方式**：观察参考图是卡片式分组还是平铺布局，还原标签对齐方式（左对齐/顶对齐）、标签与输入区的宽度比例
- **表单项样式**：从参考图提取输入框高度、圆角、边框色，以及必填标识的形式和位置
- **字段类型**：还原参考图中使用的具体表单控件类型（输入框、下拉、日期选择、开关、上传等）
- **验证反馈**：如参考图有错误状态示例，还原其提示文字位置、颜色和边框变化
- **操作栏**：还原参考图中提交/取消按钮的位置（固定底部 or 表单末尾）、按钮样式和排列顺序

### 详情页

从参考图中学习以下要素：

- **信息布局**：观察参考图中基本信息区域的列数、标签-值排列方式和间距
- **内容分区**：还原参考图中的 Tab 页签或分区方式，以及各区域的内容组织形式
- **可操作元素**：如参考图有复制按钮、编辑入口、状态标签等交互元素，忠实还原其位置和样式
- **关联数据**：还原参考图中关联表格、子列表等附属信息的展示方式

### 弹窗

从参考图中学习弹窗样式，如果参考图未包含弹窗，则根据已提取的设计令牌推算：

- **确认弹窗**：标题 + 提示文字 + 取消/确认按钮
- **危险操作弹窗**：标题带警告图标 + 确认按钮使用提取的错误状态色
- **表单弹窗**：最大宽度根据参考图推算 + 表单字段 + 取消/提交
- **抽屉弹窗**：右侧滑出，宽度 50%-80%，适合复杂表单和详情展示

## 状态标签

从参考图中学习状态标签的具体样式：

- **颜色映射**：观察参考图中不同状态（成功/警告/错误/进行中/停用等）分别使用的颜色，忠实提取色值
- **表现形式**：还原参考图中状态标签的形式（圆点+文字、色块标签、边框标签等）
- **无参考图时**：使用用户提供的品牌色推算语义色（成功=绿色系、警告=橙色系、错误=红色系）

## 操作反馈模式

- 加载状态：骨架屏、进度条、Loading 动画
- 错误处理：错误提示 + 重试机制
- 成功反馈：Toast 提示 + 状态更新
- 空状态：插图 + 提示文字 + 引导操作按钮
- 多状态展示：当参考图中同一目标物（如下拉选择器、标签、审批节点等）呈现了多个状态变体时，不仅要还原状态切换控件本身（如下拉选项），还必须在该目标物所处的列表或容器中**为每种状态至少提供一个实例卡片/行**，使开发人员能一目了然地看到各状态的视觉差异

## 使用建议

1. **有参考图时**：严格按照工作流执行——读取注册表 → 确定目标产品 → 分析参考图 → 提取设计规范 → 持久化写入 → 确定页面类型 → 交互增强 → 生成原型
2. **无参考图时**：优先使用注册表中的已有产品令牌；如无则向用户收集最低限度的设计偏好（品牌色、风格、底色），不使用任何预设设计系统
3. **状态通过交互体验**：空态/加载态/错误态通过页面右下角的「状态预览」切换器切换查看，**绝不在页面上平铺罗列**
4. **所有色值来自参考图**：不预设 body 背景色，不预设品牌色，不预设任何色值
5. **交付前**：确保所有交互效果（hover、弹窗、状态切换、搜索过滤）均可在浏览器中正常运行
6. **多产品管理**：
   - 每次会话开始时，AI 先读取注册表确定目标产品
   - 多产品场景下会提示用户选择或确认产品
   - 用户可随时说"切换到 XX 产品"来切换设计规范
   - 用户可说"查看我的产品列表"来查看所有已学习的产品
   - 用户可说"删除 XX 产品的设计规范"来清理不需要的产品
   - 用户可说"重置所有设计规范"来清空所有产品数据
